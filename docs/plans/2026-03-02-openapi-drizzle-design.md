# OpenAPI Setup and Drizzle ORM Integration Design

**Date:** 2026-03-02
**Status:** Approved
**Author:** Claude Sonnet 4.5

## Executive Summary

This design document outlines the integration of OpenAPI documentation (via Swagger UI) and Drizzle ORM into the TV Dashboards project. The approach focuses on safely migrating existing SQLite database operations to Drizzle while adding comprehensive API documentation.

## Requirements

Based on user input, the implementation must:

1. **Migrate existing tables to Drizzle ORM** - Replace raw SQL queries with type-safe Drizzle operations
2. **Full Swagger UI with interactive docs** - Install `@elysiajs/swagger` with complete API documentation
3. **Focus on existing 2 tables** - Migrate `data_source_configs` and `config_audit_log` only
4. **Full type-safe integration** - Use `drizzle-typebox` to generate Elysia validation schemas from database schema

### Migration Strategy

**Approach: Layered Migration with Backward Compatibility**

This approach was selected for safety and testability in a production environment.

**Phase 1:** Install dependencies, create Drizzle schemas matching existing tables
**Phase 2:** Add Swagger plugin to main server
**Phase 3:** Create compatibility layer that delegates to Drizzle
**Phase 4:** Update `data-source-config.js` to use Drizzle directly
**Phase 5:** Remove old `db.js` once validated

**Rationale:**
- Safe migration - existing functionality keeps working during transition
- Type safety - drizzle-typebox generates Elysia schemas from database
- Testable - can test Drizzle code alongside old code
- Reversible - can roll back if issues arise

**Trade-offs:**
- Temporary complexity with both old and new systems
- Takes longer than full replacement
- Worth it for safety in production system

### Key Design Decision

The Drizzle schema will mirror the existing SQLite schema exactly - no breaking changes to the database structure. We'll use Drizzle's introspection feature to ensure perfect compatibility, then enhance with proper TypeScript types and relations.

## Components and Data Flow

### Drizzle Schema Definition

**File:** `server/db/schema.js`

```javascript
import { sqliteTable, text, integer, index } from 'drizzle-orm/sqlite-core'
import { relations } from 'drizzle-orm'

export const dataSourceConfigs = sqliteTable('data_source_configs', {
  sourceName: text('source_name').primaryKey(),
  enabled: integer('enabled', { mode: 'boolean' }).default(true),
  configJson: text('config_json'),
  updatedAt: text('updated_at'),
  updatedBy: text('updated_by')
})

export const configAuditLog = sqliteTable('config_audit_log', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  sourceName: text('source_name'),
  action: text('action'),
  changesJson: text('changes_json'),
  userEmail: text('user_email'),
  timestamp: text('timestamp')
}, (table) => ({
  timestampIdx: index('idx_audit_timestamp').on(table.timestamp.desc()),
  sourceIdx: index('idx_audit_source').on(table.sourceName)
}))

// Define relation between tables
export const dataSourceConfigsRelations = relations(dataSourceConfigs, ({ many }) => ({
  auditLogs: many(configAuditLog)
}))
```

**Key Features:**
- Exact column mapping to existing schema
- Proper type conversions (integer boolean mode for enabled)
- Index definitions match existing database
- Relations defined for potential joins

### Drizzle Instance Setup

**File:** `server/db/index.js`

```javascript
import { drizzle } from 'drizzle-orm/bun-sqlite'
import { Database } from 'bun:sqlite'
import * as schema from './schema.js'

const sqlite = new Database('data/tv-dashboards.db')
export const db = drizzle(sqlite, { schema })

// Export schemas for use in queries
export { dataSourceConfigs, configAuditLog } from './schema.js'
```

**Design Notes:**
- Uses `bun:sqlite` (same as existing code)
- Exports schema for type-safe queries elsewhere
- Single database instance (singleton pattern)

### Data Flow

```
Request → Elysia Route Handler
         ↓
      Validation (Elysia schema from drizzle-typebox)
         ↓
      Database Query (Drizzle ORM - type-safe)
         ↓
      Response → Auto-documented via Swagger
```

### Type Safety Chain

1. **Drizzle schema** defines database structure
2. **drizzle-typebox** generates Elysia validation schemas
3. **TypeScript** infers types throughout the stack
4. **Swagger** auto-generates API documentation

This creates a single source of truth from database → validation → documentation.

## Error Handling

### Database Error Handling

Drizzle queries will be wrapped with try-catch blocks and proper error translation:

```javascript
// Example in data-source-config.js
import { db, dataSourceConfigs } from './db/index.js'
import { eq } from 'drizzle-orm'

export function getConfig(sourceName) {
  try {
    const result = db.select()
      .from(dataSourceConfigs)
      .where(eq(dataSourceConfigs.sourceName, sourceName))
      .get()

    if (!result) {
      throw new Error(`Data source "${sourceName}" not found`)
    }
    return result
  } catch (error) {
    logger.error({ sourceName, error: error.message }, 'Failed to get config')
    throw error
  }
}
```

**Error Handling Strategy:**
- Preserve existing error messages and behavior
- Add structured logging with context
- Maintain existing HTTP status codes
- Clear error messages for debugging

### Migration Error Handling

The migration runner will:

- Check database connectivity before running migrations
- Validate schema integrity after migrations
- Log all migration steps
- Provide rollback instructions if migration fails
- Exit cleanly on error without corrupting the database

### Swagger Error Responses

API documentation will include proper error response schemas:

```javascript
.get('/api/data-sources/:name/config', ({ params }) => {
  // handler
}, {
  detail: {
    tags: ['data-sources'],
    responses: {
      200: { description: 'Config retrieved successfully' },
      404: { description: 'Data source not found' },
      500: { description: 'Server error' }
    }
  }
})
```

### Backward Compatibility Errors

During the transition phase, if Drizzle queries fail:

1. Log the error with full context
2. Do NOT fall back to old code (to catch issues early)
3. Return clear error messages indicating migration issues
4. Preserve existing error handling behavior for API consumers

**Rationale:** Failing fast during migration helps identify issues immediately rather than silently falling back and hiding problems.

## Testing Strategy

### Database Migration Testing

**1. Schema Validation Test:**
- Compare Drizzle-generated schema with existing SQLite schema
- Verify all columns, types, indexes match exactly
- Use `drizzle-kit introspect` to validate
- Run before and after migration

**2. Data Integrity Test:**
- Query existing data with both old and new code
- Verify results are identical
- Test with actual production database snapshot
- Compare row counts, data types, JSON parsing

### Drizzle Query Testing

Create unit tests for each migrated database operation:

```javascript
// tests/unit/db-drizzle.test.js
import { describe, test, expect, beforeEach } from 'bun:test'
import { db, dataSourceConfigs } from '../server/db/index.js'
import { eq } from 'drizzle-orm'

describe('Drizzle Data Source Config', () => {
  test('should retrieve config by source name', () => {
    const result = db.select()
      .from(dataSourceConfigs)
      .where(eq(dataSourceConfigs.sourceName, 'gcp'))
      .get()

    expect(result).toBeDefined()
    expect(result.sourceName).toBe('gcp')
  })

  // More tests for insert, update, delete operations
})
```

**Test Coverage:**
- SELECT operations (single, multiple, with filters)
- INSERT operations (single, batch)
- UPDATE operations (single, conditional)
- DELETE operations
- Transaction handling
- Error cases

### Swagger Documentation Testing

**1. Schema Generation Test:**
- Verify `/swagger/json` endpoint returns valid OpenAPI 3.0 spec
- Check all routes are documented
- Validate schema definitions match Drizzle types
- Ensure response examples are accurate

**2. UI Accessibility Test:**
- Access `/swagger` endpoint
- Verify Swagger UI renders correctly
- Test "Try it out" functionality on key endpoints
- Validate authentication flows (if applicable)

### Integration Testing

Test the full stack with Drizzle + Swagger:

```javascript
// tests/integration/api-with-drizzle.test.js
```

**Integration Test Scenarios:**
- End-to-end API request with database query
- Error responses match Swagger documentation
- Validation errors return proper status codes
- Complex queries with relations work correctly

### Manual Testing Checklist

Before considering migration complete:

- [ ] Swagger UI loads at `/swagger`
- [ ] All API endpoints appear in documentation
- [ ] "Try it out" works for GET endpoints
- [ ] Schema validation works for POST/PUT endpoints
- [ ] Database queries return correct data
- [ ] Audit log entries are created properly
- [ ] Error responses match documentation
- [ ] Performance is acceptable (no regressions)

### Migration Validation

Before removing old `db.js`:

1. Run all existing tests - they should pass
2. Compare query performance (Drizzle should be similar or faster)
3. Verify no console errors in production
4. Check database file size hasn't changed
5. Review audit logs for any anomalies
6. Test rollback procedure

## Implementation Phases

### Phase 1: Setup and Schema Creation
- Install dependencies (`drizzle-orm`, `drizzle-kit`, `@elysiajs/swagger`, `drizzle-typebox`)
- Create `server/db/schema.js` with table definitions
- Create `drizzle.config.js` for Drizzle Kit
- Run `drizzle-kit introspect` to validate schema matches existing database

### Phase 2: Swagger Integration
- Add `@elysiajs/swagger` plugin to main server
- Configure documentation metadata
- Add tags to existing routes
- Test Swagger UI accessibility

### Phase 3: Drizzle Instance and Migration
- Create `server/db/index.js` with Drizzle instance
- Create `server/db/migrate.js` for migration runner
- Generate initial migration with `drizzle-kit generate`
- Test migration on development database

### Phase 4: Query Migration
- Update `data-source-config.js` to use Drizzle queries
- Preserve exact same API surface (function signatures)
- Add comprehensive unit tests
- Run integration tests

### Phase 5: Validation and Cleanup
- Run full test suite
- Manual testing of all database operations
- Performance testing
- Remove old `db.js` once validated
- Update documentation

## Success Criteria

The implementation is considered successful when:

1. All existing tests pass with Drizzle queries
2. Swagger UI displays complete API documentation at `/swagger`
3. Type safety is enforced (TypeScript/JSDoc with no `any` types)
4. Database queries return identical results to old implementation
5. No performance regressions (queries complete in similar time)
6. All manual testing checklist items are verified
7. Code review passes and PR is merged

## Risks and Mitigations

### Risk: Data Loss During Migration
**Mitigation:**
- No data migration needed (Drizzle uses existing database)
- Automated backups before any schema changes
- Test on database snapshot first

### Risk: Performance Regression
**Mitigation:**
- Drizzle generates efficient SQL
- Keep existing indexes
- Benchmark before/after migration

### Risk: Breaking Changes to API
**Mitigation:**
- Preserve exact function signatures
- Maintain same error handling behavior
- Integration tests verify API contracts

### Risk: Incomplete Documentation
**Mitigation:**
- Review Swagger UI for all endpoints
- Add missing tags/descriptions
- Validate against OpenAPI spec

This design provides a safe, incremental approach to integrating Drizzle ORM and OpenAPI documentation into the TV Dashboards project. The layered migration strategy minimizes risk while delivering improved type safety, better developer experience, and comprehensive API documentation.

The focus on the two existing tables keeps the scope manageable while establishing patterns that can be extended to future database work.
