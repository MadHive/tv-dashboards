## Component Inventory

All components extend existing DOM patterns ‚Äî no new component files created outside the established file structure.

### Query Builder Panel (QRYX-01, QRYX-02)

**Widget-scoped flow (right panel slot):**
- Entry: "Build Query" button added to the "Data" section of the properties panel
  - Style: `.studio-btn.secondary.small` ‚Äî matches existing secondary small buttons in properties panel
  - Label: "Build Query"
- Panel opens in the right `#query-editor-panel` slot (replaces properties content via `style.display` toggle ‚Äî existing pattern)
- Panel header: shows query name (editable inline `<input>`) + source badge (`.qe-source-badge`)
- Config row: Source `<select>` + metric type `<input>` ‚Äî uses existing `.qe-label`, `.qe-select`, `.qe-input`
- Action row: "Run Query" button (`.studio-btn.primary.small`) + "Cancel" button (`.studio-btn.ghost.small`)
- Result area: `.qe-results` container (existing class) ‚Äî table for structured data, pre-formatted JSON for raw results, summary stats row for time-series (3 labels: data points, time range, last value)
- Result truncation: 100 rows max for table; display truncation notice as `.results-truncated` text below table ("Showing first 100 rows")
- After successful run: "Assign to Widget" button appears below results (`.studio-btn.primary`, full width of panel) ‚Äî sets widget `queryId` + `source`, closes panel, shows success toast

**Freestanding flow (Queries sidebar tab):**
- Entry: "+ New Query" button in the Queries sidebar tab header (`.sidebar-add-btn`)
- Opens a centered modal (`.query-editor-modal` / `.query-editor-container` ‚Äî reuse existing `query-editor.css` modal) at 95vw √ó 90vh
- Modal header: "New Query" title (20px, weight 600, `--font-display`) + close button
- Modal body: sidebar listing existing queries (`.query-editor-sidebar`) + main editor area (`.query-editor-main`)
- Modal save: "Save Query" button (`.btn.btn-primary`) ‚Äî saves to library, dismisses modal

### Data Source Health Section (DSRC-01)

- Location: top of Sources sidebar tab, above existing source list
- Rendered as a `.sidebar-section` with `.sidebar-section-header.collapsible`
- Section label text: "HEALTH" (uppercase, 10px, weight 600, letter-spacing 2px ‚Äî matches all other section headers)
- Default state: collapsed (`display:none` on content)
- Expanded state: shows a compact grid of source health rows ‚Äî one row per data source
- Per-source health row layout (horizontal, 14px height):
  - Status dot (`.ds-status-dot` 8px circle) ‚Äî green=connected, amber=mock, red=disconnected, grey=unknown
  - Source name (13px, `--font-body`, `--t2`)
  - Last-success timestamp (10px, `--font-mono`, `--t3`, right-aligned, format: "5m ago" / "2h ago" / "never")
  - Error count badge (only shown when > 0): red pill with count, 10px mono
- On click of a source row: expands inline to show last error message (12px, `--font-mono`, `--red`, 2-line clamp with "show more" link)
- Auto-refresh: fetch `GET /api/data-sources/health` on tab focus + every 30s while Sources tab active
- Polling indicator: animated `--mh-pink` dot in section header (pulse animation, 6px) while auto-refresh is running

### Credential Validation Flow (DSRC-02)

- Validation is triggered on "Save Credentials" button click (existing button, no layout change)
- Step 1 ‚Äî Client-side format check: fields validate non-empty + basic structure before fetch; errors shown inline below each field as `.validation-error` text (12px, `--red`, `--font-body`)
- Step 2 ‚Äî Server-side: on format pass, call `PUT /api/data-sources/:name/credentials`; if server rejects, render error banner at top of credential form (`.validation-error` div, full width, 12px, `--red`)
- Success: `showToast('Credentials saved', 'success')` ‚Äî no separate success UI beyond toast
- Save button disabled state during in-flight request: `opacity: 0.3`, `cursor: not-allowed` (`.studio-btn:disabled`)

### Unified Metric Browser (METR-01)

- Extends existing `MetricBrowser` class and `#metric-browser-modal`
- Source selector: tab strip added above the existing GCP service sidebar
  - Tabs: "GCP" | "BigQuery" | "VulnTrack" + any `isConnected=true` sources
  - Tab style: matches existing `.query-tab` pattern (10px uppercase label, active tab gets pink `border-bottom: 2px solid var(--mh-pink)`)
  - Disconnected sources: shown greyed out (`--t3`, `cursor: not-allowed`, no click)
- GCP tab: existing service sidebar + metric list (unchanged)
- BigQuery tab: static manifest of known `mad-data` tables rendered as a searchable list (same search input as GCP tab, same row pattern)
- VulnTrack tab: static curated list of VulnTrack metric endpoints
- Assignment: selecting a metric closes the browser and assigns `source` + `queryId` to the widget ‚Äî same behavior as existing GCP direct-assign

## Phase-Specific Interaction Rules

These rules apply only to Phase 3 and are not in scope for the shared studio pattern library:

1. **Query editor panel exclusivity:** Opening the query builder from the properties panel replaces the properties content; returning is via "Cancel" button or clicking a different widget on canvas. The panel toggle mechanism is `style.display` ‚Äî the same as existing right-panel panels.

2. **Result format selection:** The result preview format is chosen automatically by the executor based on data shape ‚Äî table for arrays of objects, `<pre>` block for raw JSON, summary stats for time-series (point count + time range + last value). No user toggle required.

3. **Health section polling lifecycle:** Polling starts when the Sources tab becomes active (`tabclick` event), stops when any other tab is selected. Uses `setInterval` / `clearInterval` ‚Äî no WebSocket.

4. **Metric browser source tabs:** Only tabs for `isConnected=true` sources are rendered (plus GCP always). The tab list is built dynamically from `GET /api/data-sources` at modal open time.

5. **Assign flow closes panel/modal:** After "Assign to Widget" or metric browser assignment, the panel/modal closes and `renderCanvas()` is called to reflect the updated widget data source. A success toast fires: `showToast('Query assigned', 'success')`.
